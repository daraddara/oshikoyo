# バージョン管理規定

## 基本方針

おしこよは [Semantic Versioning 2.0.0](https://semver.org/lang/ja/) に準拠し、`MAJOR.MINOR.PATCH` 形式でバージョンを管理します。

---

## バージョン番号の意味

| セグメント | 記号 | 意味 |
|-----------|------|------|
| MAJOR     | `X`  | 後方互換性のない破壊的変更（データ構造の非互換変更、設計の抜本刷新など） |
| MINOR     | `Y`  | 後方互換性を保った機能追加・UI刷新 |
| PATCH     | `Z`  | バグ修正・軽微な改善（機能変化なし） |

---

## フェーズ定義

### 開発試作フェーズ（`0.Y.Z`）

- **MAJOR = 0** の間は開発試作中です。
- 公開 API・データ構造・UI は固定されておらず、MINOR バージョンアップでも破壊的変更を行うことがあります。
- `0.Y` の Y は **機能追加・改修イテレーション**を表します（旧内部ビルドカウンタ v1〜v28 を `0.28` に引き継ぎ）。

### 正式リリース（`1.0.0`〜）

- 機能・UX・データ構造が安定し、一般向けに公開できると判断した時点で `1.0.0` を宣言します。
- 以降は MAJOR の破壊的変更を慎重に行い、ユーザーへの影響を最小化します。

---

## バージョンの正

| ファイル | 記述 | 役割 |
|---------|------|------|
| `package.json` の `"version"` | `"1.0.0"` | **唯一の正（Single Source of Truth）** |
| `src/script.js` の `APP_VERSION` | `'v1.0.0'` | UI 表示用。package.json と常に一致させる |

> **規則:** `package.json` を先に更新し、`APP_VERSION` をそれに合わせて変更してください。
> 二重管理によるバージョン不整合を防ぐため、`version-badge` の表示は JS が動的に挿入します（HTML への直書き禁止）。

---

## バージョンアップのタイミング

| 状況 | 変更箇所 | 例 |
|------|---------|-----|
| バグ修正・文言修正・スタイル微調整 | PATCH を上げる | `0.28.0` → `0.28.1` |
| 新機能追加・UI 刷新・設定項目追加 | MINOR を上げ、PATCH を 0 にリセット | `0.28.0` → `0.29.0` |
| データ構造・ストレージ仕様の破壊的変更 | MINOR を上げる（0.x.x フェーズは MINOR で破壊的変更を許容） | `0.28.0` → `0.29.0` |
| 正式リリース宣言 | MAJOR を 1 に上げ、MINOR・PATCH を 0 にリセット | `0.28.x` → `1.0.0` |

---

## バージョンアップ手順

1. `package.json` の `"version"` を更新する
2. `src/script.js` の `APP_VERSION` を `'v<同じ値>'` に更新する
3. `sw.js` の `CACHE_NAME` を `'oshikoyo-v<同じ値>'` に更新する（既存PWAユーザーへの配信に必須）
4. `npm test` を実行し全テストのパスを確認する
5. コミットメッセージに `chore: bump version to vX.Y.Z` を含める

```bash
# 例: 1.0.0 → 1.0.1
# 1. package.json の "version": "1.0.0" を "1.0.1" に変更
# 2. script.js の APP_VERSION = 'v1.0.0' を 'v1.0.1' に変更
# 3. sw.js の CACHE_NAME = 'oshikoyo-v1.0.0' を 'oshikoyo-v1.0.1' に変更
# 4. npm test
# 5. git commit -m "chore: bump version to v1.0.1"
```
