# タグ機能 実装計画レビュー

> 対象ドキュメント: [tag-feature-plan.md](./tag-feature-plan.md)

---

## 全体評価

**良い点**

- シンプルな `string[]` 構造の選択は正しい。タグに色・階層・メタデータを持たせると複雑化が加速するので、この判断は維持すべき。
- 段階的なフェーズ分けの考え方は適切。

---

## 懸念点・提案

### 1. 画像タグの格納場所（最重要）

計画では「IndexedDB の画像レコードに `tags: string[]` を追加」とあるが、**IndexedDB のスキーマ変更は避けることを強く推奨する**。

**理由:**
- IndexedDB のスキーマ変更は `onupgradeneeded` での `version` 管理が必要で、既存データの移行コストが高い
- `appSettings.localImageOrder` と同じく、**`appSettings.localImageMeta`** に格納すれば一貫性が保てる
- 設定エクスポート（`oshikoyo_settings_*.json`）に自動で含まれる
- バックアップ・リストアの単純さが維持される

**提案する構造:**

```js
// appSettings への追加
localImageMeta: {
    42: { tags: ['誕生日', '衣装A'] },
    57: { tags: ['ユニットX'] },
}
```

> **注意:** 画像を別ブラウザに移植する場合、IndexedDB の ID が変わるため tags も引き継げない。
> ただし、settings + images のセットエクスポート運用で対応できる範囲として許容。

---

### 2. oshi の `tags` の命名

計画では `appSettings.oshis[].tags` と記載があるが、現在のコードは **`appSettings.oshiList`** である。
混乱を防ぐため `appSettings.oshiList[].tags` に揃えること。

---

### 3. Phase の順序について

**現在の順序:** Phase 1（管理UI）→ Phase 2（登録時プレビュー付与）→ Phase 3（フィルター）

**提案:** Phase 2 を Phase 1 の直後に前倒しすることを検討。

登録時にタグを付けられないと、Phase 1 の「一覧での閲覧・編集」機能だけでは使い始めにくい。
「登録 → タグ付け → 整理」の動線を早期に確立した方が、フィードバックも得やすい。

## オープン質問への見解

### ① 複数画像の同時 D&D + タグ入力 UX

**「全体に一括付与」＋「個別上書き可」のハイブリッドを推奨する。**

```
┌─────────────────────────────────┐
│ 全画像に共通タグを付与（任意）           │
│ [誕生日] [衣装A] [+追加...]            │
│                                 │
│ [img1]   [img2]   [img3] ...    │
│  ＋個別タグ  ＋個別タグ  ＋個別タグ    │
└─────────────────────────────────┘
```

枚数が多い（10枚超など）場合、1枚ずつカルーセルで確認させるのは UX として重すぎる。
共通タグだけ付けて保存できる「逃げ道」を設けることが重要。

### ② グリッド上のタグ視覚表現

88px タイルにタグを表示しようとするとノイズになる。

- **基本方針:** ライトボックス（個別表示）でのみ確認・編集
- **グリッド上:** タグがある画像には小さなインジケーター（角のドット等）だけ表示

### ③ 記念日フィルター連動アルゴリズム

**重み付け（ソフトフィルター）一択。**

- 対象タグの画像を高確率で引き当て、なければ通常の pool から選ぶ
- 厳格フィルターだと「誕生日なのに画像が表示されない」という最悪の体験が生じる

## 原案者確認事項

### タグのマスターリスト管理方針（確認済み）

- 新しいタグが手入力された際、**自動で `appSettings.tags` に追記・永続化する**
- 専用のタグ管理・削除画面は Phase 1 では作らない
- 不要タグの削除機能は後回しでよい

> **将来の削除機能についての補足:**
> 手動管理画面より、「どの画像・推しにも紐づいていないタグを一括削除」する自動クリーンアップの方が実装コストが低くシンプル。

---

## まとめ

| 項目 | 判断 |
|---|---|
| シンプル `string[]` | 維持 |
| 画像タグの格納先 | IndexedDB ではなく `appSettings.localImageMeta` に変更 |
| oshi の命名 | `oshiList[].tags` に統一 |
| Phase 順序 | Phase 2 を前倒し検討 |
| 複数画像 UX | 一括共通タグ ＋ 個別上書きのハイブリッド |
| グリッド表示 | インジケーターのみ、詳細はライトボックス |
| 記念日アルゴリズム | 重み付け（ソフトフィルター） |
