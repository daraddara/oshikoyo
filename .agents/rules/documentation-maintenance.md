# ドキュメント管理の原則
- **唯一の真実 (Single Source of Truth)**: 
  - コーディング規約、Git運用、テストの具体的な実行手順などの「技術的強制力を持つルール」は `.agent/rules/` 内にのみ記述してください。
  - ルートディレクトリの Markdown ファイル（README, TESTING, DEVELOPMENT等）には、上記ルールの具体的なコード例や手順を重複して記載しないでください。

# 各ファイルの保守指針
1. **README.md**: プロジェクトの目的、主要機能、技術スタックの概要に限定します。詳細な開発手順は `DEVELOPMENT.md` やエージェントルールへ誘導してください。
2. **DEVELOPMENT.md**: 環境構築（npm install等）の初歩的な手順のみを残します。Gitのブランチ命名規則などは `.agent/rules/git_branching_policy.md` に一元化し、このファイルからは削除または参照のみに留めてください。
3. **TESTING.md**: テストの理念と対象（何をテストするか）を記述します。具体的な実行コマンドや、特殊なコード抽出ロジック（new Function等）の詳細はエージェントルールに任せ、重複を削除してください。
4. **ISSUES.md**: 開発中に判明した仕様の曖昧点や、未着手の機能案を常に最新に保ってください。ワークフロー（`/feature_finish`等）で課題が解決された場合は、自律的にステータスを更新してください。

# 自動更新ルール
- 機能を新規追加、または仕様変更した際は、必ず `README.md` の「特徴」セクションを更新してください。
- リファクタリングによって仕様の曖昧さが解消された場合は、`ISSUES.md` から該当項目を削除するか「完了」に移動してください。

# 報告書（Walkthrough）の画像パスに関する規則
- **絶対パスの禁止**: Walkthrough 内で画像を参照する際、OSの絶対パス（`C:\` や `/Users/` で始まる形式）は決して使用しないでください。
- **相対パスの原則**: プロジェクトルートからの相対パス、またはブラウザツールが提供する「アーティファクト相対パス（例: `./screenshots/xxx.png`）」のみを使用してください。
- **試行錯誤の制限**: パス修正で2回以上エラーが発生した場合は、強引にパスを推測し続けず、画像なしでテキストのみの報告を作成することを優先し、ユーザーにパスの問題を報告してください。
