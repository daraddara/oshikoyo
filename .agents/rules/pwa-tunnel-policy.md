# 目的
スマートフォン等の実機で「おしこよ」のPWAとしての動作（ホーム画面への追加、Service Workerの動作、オフラインキャッシュ、レイアウト等）を検証するために、ローカルサーバーを外部公開する際の手順とエージェントの責務を定義します。

# 基本方針

## 1. 責務の分離
- **エージェントの責務**:
  - `npm run serve` を用いてローカルサーバーを起動し、正常動作（ポート8081の待機）を保証すること。
  - `npm run tunnel`（`cloudflared tunnel --url http://localhost:8081`）を実行し、一時的な公開URL（`https://*.trycloudflare.com`）を発行・抽出すること。
  - プロセス全体のライフサイクル管理および終了時のクリーンアップを行うこと。
- **ユーザーの責務**:
  - 発行されたURLに対して、手元のスマートフォン等の実機ブラウザからアクセスし、UIやPWAマニフェストを含む実際の動作確認を行うこと。

## 2. 起動手順とURLの抽出
- トンネルを利用する際（例：`/pwa` ワークフローの実行時）は、**必ず `scripts/pwa-serve.sh` を使用してください**。個別コマンドを直接実行することは禁止します。
  ```bash
  ./scripts/pwa-serve.sh
  ```
- スクリプトがサーバー起動確認・URL抽出・表示を一括で行います。スクリプトの出力からURLをユーザーに提示してください。
- `scripts/pwa-serve.sh` の内容（ポート番号・待機時間・ログパス等）が実態と乖離した場合は、使用前にスクリプトをメンテナンスしてから実行してください。

## 3. クリーンアップとプロセス管理
- トンネル公開が不要になった時点、もしくはユーザーからの終了指示があった場合には、起動した `serve` (Node.js) と `cloudflared` を両方とも完全に終了させてください。
- 終了漏れが原因で次回起動時にポート競合やエラーが発生しないよう、必要に応じて `.agents/rules/execute-tests.md` に記載のプロセス終了手順に準じたクリーンアップ（`npx kill-port 8081` や必要なプロセスの終了）を行ってください。

# Service Worker キャッシュバージョン管理ポリシー

## バージョンの役割

`sw.js` の `CACHE_NAME`（例: `oshikoyo-v13`）はブラウザ内部のキャッシュキーであり、
ユーザーに表示されるアプリバージョンとは無関係です。
値が変わると古いキャッシュが破棄され、全キャッシュ対象ファイルが再取得されます。

## バージョンを上げるタイミング（必須条件）

以下のファイルを変更し、**ユーザーの手元に確実に届ける必要がある**場合のみ上げてください:

- `src/style.css` / `src/script.js` / `index.html`
- `sw.js` 自体
- `manifest.json` などPWAマニフェスト

コミット時に `sw.js` のバージョンアップを忘れた場合、ユーザーが古いキャッシュを参照し続けます。

## バージョンを上げてはいけないタイミング

- 調査・デバッグ中（「キャッシュが原因かもしれない」という推定のみの場合）
- テストファイル・ドキュメントのみの変更
- CSS/JSを変更していないコミット

## 開発中のキャッシュ確認方法

バージョンを上げずにキャッシュを無効化したい場合は、DevTools を使用してください:

- **Application → Service Workers → "Update on reload"** をオン
- またはDevTools → Network → **"Disable cache"** をオン（DevTools 開放中のみ有効）

## バージョン番号のリセットについて

リセットは不要です。番号に意味はなく、「前回と異なる文字列か」だけが重要です。
正式リリース時も現在の連番を継続してください。
