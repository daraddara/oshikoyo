# PR管理の原則
- PRの処理は、常に「現状分析」「ローカル検証」「競合解消」「マージ」の4ステップで行います。
- `gh` (GitHub CLI) コマンドを活用し、ブラウザを介さずターミナル内で完結させることを推奨します。

# 1. 現状分析と対象の特定
- まず `gh pr list` を実行し、オープンなPRの一覧を表示します。
- 複数のPRが存在する場合、処理対象の **PR番号** を必ず特定してください。
- 特定のPRの内容を確認・議論する場合は、`gh pr view [PR番号]` を使用します。

# 2. ローカル検証
- 対象のPRは、必ず `gh pr checkout [PR番号]` を使用してローカルに取得してください。
  - `[PR番号]` は `gh pr list` で確認した番号、またはURL/ブランチ名を指定します。
- 取得後、`npm test` および `npm run e2e` を実行し、既存機能への影響がないか、およびPRの内容が正しく動作するかを確認します。
- ブラウザでの目視確認が必要な場合は、ポート `8081` を使用して検証を行い、エビデンス（スクリーンショット）を取得してください。

# 3. マージコンフリクトの解消
- PRが `main` ブランチとコンフリクトしている場合、自動マージは行わず、以下のいずれかの手順で手動解消してください：
  - **Rebase**: `git fetch origin main` の後、`git rebase origin/main` を行い、コンフリクトを一行ずつ解消します。
  - **Merge**: `git fetch origin main` の後、`git merge origin/main` を行います。
- 解消後は、再度テストを行い、整合性が保たれていることを確認してください。

# 4. マージの実施
- `git-branching-policy.md` の「マージの条件 (Definition of Done)」をすべて満たしていることを再確認します。
- マージは原則として **Squash Merge** 行います。
- コマンド例: `gh pr merge --squash --delete-branch`
  - `--delete-branch` フラグを使用することで、マージ完了と同時にリモートブランチが削除され、不要なブランチが残るのを防げます。

# 注意事項
- PR作業中に大規模なリファクタリングを勝手に含めないでください。必要な場合は別途PRを作成してください。
- PRの取得・作業を行う前には、必ずローカルの `main` ブランチが最新であることを確認してください。

# ネットワーク通信・APIコマンド実行時の特則（二重実行防止）
- `gh pr comment` や `git push` など、外部サーバーとの通信を伴うコマンドを実行した際は、**必ずプロセスの完了（DONE）を十分に待機**してください。
- **ハングアップの誤認防止**: 5〜10秒程度で終わらなくても失敗と即断してはいけません。最低でも30秒〜60秒は `command_status` で待機し、確実に成否を見届けてください。
- **強制終了の禁止**: 待機中のジョブに対してシェルから無効な `kill` コマンドで強制終了を試みることは厳禁です（バックグラウンドで処理が継続し、二重実行の原因になります）。
- **再試行前の状態確認（冪等性）**: タイムアウト等でコマンドが失敗し再試行を行う場合は、直前に `gh api` や `gh pr view` 等を使用して、「前回のコマンド結果がサーバー側に反映されていないか」を必ず確認してください。
