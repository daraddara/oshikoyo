# コマンド実行の標準手順 (Command Execution SOP)

このルールは、エージェントがターミナルコマンドを実行する際の「不必要な待機」と「不透明な停止（ハングアップ）」を完全に排除し、ユーザー体験を損なわないための規定です。

## 1. ブロッキング待機の禁止
- **最大待機時間の制限**: `command_status` ツールの `WaitDurationSeconds` に `300` などの大きな値を設定することを**厳禁**します。
- **推奨値**: 原則として `0`〜`10` 秒程度に設定し、可能な限り `0` で即時レスポンスを返してください。
- **理由**: 長時間の待機を設定すると、プロセスが SSH パスワードや GPG パスフレーズなどの「標準入力（プロンプト）」を待って停止している場合、エージェントが数分間黙り込んでしまい、ユーザーからは「フリーズしている」ように見えるためです。

## 2. 非対話モード（Non-Interactive）の徹底
通信を伴うコマンド（git, gh, npm 等）を実行する際は、プロンプトが表示されて停止しないよう、必ず以下の対策を講じてください。

- **Git の非対話化**: `GIT_TERMINAL_PROMPT=0` や `GIT_SSH_COMMAND="ssh -o BatchMode=yes"` を環境変数に付与し、パスワード入力等を即座にエラーにします。
- **GitHub CLI (gh)**: `gh` コマンドには対話的なプロンプトを避けるためのフラグ（`--non-interactive` や `--yes` 等）を付与するか、環境変数 `GH_PROMPT_DISABLED=1` を使用してください。
- **npm**: インストールや実行時に `--yes` などを付与し、確認ダイアログを回避します。

## 3. 異常検知とプロアクティブな報告
- **プロセスの監視**: `command_status` で `RUNNING` 状態が続き、かつ標準出力に変化が数秒〜十数秒以上見られない場合は、ハングアップを疑ってください。
- **報告の義務**: 何らかの理由でコマンドが長時間（例: 2分以上）かかる場合は、黙って待つのではなく、必ずその理由と現在の出力をユーザーに報告し、継続か中断かを相談してください。
- **強制終了の判断**: プロンプト待ちが明らかな場合やタイムアウトした場合は、いたずらに再試行せず、一旦プロセスを終了させてエラーの原因を分析してください。

## 4. SSH/ネットワーク接続の特則
- **IdentityAgent などの考慮**: WSL やリモート環境で SSH エージェント転送に関わる遅延が発生しやすい場合は、`IdentityAgent=none` や `ConnectTimeout` オプションを明示的に使用して、接続トラブルを早期に表面化させてください。
