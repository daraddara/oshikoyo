# ブランチ戦略
- `main` ブランチへの直接の修正・コミットは禁止します。
- 新しい機能開発や修正を行う際は、必ず目的に応じた新しいブランチを作成してください。
- 命名規則:
  - 新機能: `feature/[機能名]`
  - バグ修正: `fix/[問題名]`
  - リファクタリング: `refactor/[対象]`

# 作業開始時の強制手順
- いかなるファイルの編集（書き換え）を開始する前にも、必ず現在のブランチを確認してください。
- 現在のブランチが `main` または `master` である場合、編集を一行も行う前に、直ちに適切な作業ブランチを作成し、切り替えてください。
- 「まず編集してから後でブランチを作る」という手順は、競合のリスクがあるため厳禁です。
- **フェーズ移行時の注意**: `/plan` による検討や企画フェーズから実装（EXECUTION）に移行する際は、思考の慣性で `main` のまま作業を続けがちです。ファイルを一行でも触る前に、必ず `git checkout -b` を実行してください。

# コミットとマージの条件
- コミット前に、必ず `npm test` を実行し、すべてのテストがパスしていることを確認してください。
- コミットメッセージは [Conventional Commits](https://www.conventionalcommits.org/) 形式に従ってください。
- 作業完了後はユーザーに報告し、マージの許可を得てください。

# 自己マージの禁止
- **PR を作成したエージェントは、自分の PR を自分でマージしてはなりません。**
- PR 作成後は、別エージェントまたはユーザーにレビュー・マージを委ねてください。
- 例外（緊急対応等）は `docs/AGENTS.md` セクション10の「例外」条件を参照してください。

# マージの条件 (Definition of Done)
- 以下の条件をすべて満たさない限り、`main` ブランチへのマージを提案・実行しないでください:
  1. すべてのロジックテスト（Vitest）が 100% パスしていること。
  2. 修正したコードに対し、`communication_style.md` に従った JSDoc 等のドキュメントが付与されていること。
  3. `ISSUES.md` に関連する修正の場合、該当項目のステータスが更新されていること。
- マージ前には、必ず `git diff main` を行い、意図しない変更が含まれていないか自己監査してください。

# PRレビュー時の制約
- **他者のPRに対する直接修正の原則禁止**: 
  - コンフリクトの解消を除き、レビュー対象のPRブランチを直接編集（commit/push）することは禁止します。
  - 問題を発見した場合は、自ら修正するのではなく、GitHubのコメント機能や修正提案（Suggested Changes）を使用して作者にフィードバックを行ってください。
- **対応方針の相談**:
  - 重大なバグや仕様の誤りがある場合でも、まずはコメントで指摘し、作者またはユーザーと対応方針（作者による修正か、エージェントによる肩代わりか）を協議してください。
  - エージェントが直接修正を行うのは、ユーザーから明示的な指示があった場合、または「極めて軽微なタイポ修正」などの例外ケースに限られます。


# マージ戦略と履歴の管理
- **マージ方式**: 常に **Squash Merge** を使用してください。複数のコミットを一つの意味のあるコミットにまとめ、マージコミットを作成せずに `main` へ統合します。
- **リニア履歴の維持**: `main` ブランチの履歴は常に一本道（Linear History）であることを確認してください。
- **コミットメッセージ**: Squash Merge 時のメッセージは、そのブランチで行った変更の要約（例: `feat: 祝日ロジックの保守性を向上 (ISSUES A-01)`）としてください。
- **Force Push の禁止**: `main` ブランチへの `git push --force` / `git push -f` はいかなる場合も禁止です。履歴の書き換えが必要な場合はユーザーに相談してください。

# ブランチの削除
- PRがマージまたはクローズされた後は、速やかに作業ブランチを削除してください。
- リモートブランチの削除には `gh pr merge --delete-branch` または `git push origin --delete [ブランチ名]` を使用します。
- ローカルブランチは `git branch -d [ブランチ名]` で削除し、リポジトリの肥大化を防いでください。
- 定期的に `/cleanup` ワークフローを実行し、マージ済みの不要なブランチを一掃することを推奨します。
