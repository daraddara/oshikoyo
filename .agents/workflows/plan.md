# 目的
本格的な要件定義や技術設計に入る「前」の段階で、ユーザーとアイディアの壁打ちを行い、他の類似アプリやトレンドを参考にしながら実現可能性と方向性を探るためのワークフローです。

# 手順
1. **アイディアのヒアリングと深掘り**:
   - ユーザーが実現したいこと（What）と、その背景にある課題や目的（Why）をまずは丁寧に聞き出します。
   - 足りない視点や、さらに面白くなりそうなアイディアがあれば、積極的に提案して議論を広げてください。

2. **外部調査・他アプリの事例参照**:
   - ユーザーから他アプリの事例について言及があった場合や、UI/UXの参考が必要な場合は、自律的に `search_web` 等を用いて類似アプリ（推し活アプリやカレンダーアプリ等）のトレンドや機能事例を調査します。
   - 調査結果を元に、このアプリ（おしこよ）にどう組み込めるか提案してください。

3. **実現可能性のブレインストーミング**:
   - 現在の技術スタック（Vanilla JS, CSS, HTML, LocalStorageなど）で実現可能か、ざっくりとした検討を行います。
   - 実装の難易度や、将来的な拡張性について、専門的な視点からアドバイスを提供します。

4. **具体案（モックアップ・UIイメージ）の提示**:
   - 必要に応じて、HTML/CSSの簡易モックコード、テキストベース of UIレイアウト案、または画像生成ツールによるUIイメージの提案を行い、お互いの想像を膨らませます。
   - **この段階では実際のプロジェクトソースコード（.jsや.css、HTML等の本番ファイル）の変更機能の実装は行いません。** あくまで「検討」に留めます。

5. **ネクストステップの合意とドキュメント化**:
   - アイディアが固まったら、それを具体的な Issue として `ISSUES.md` に要件を追記することを提案または代行します。
   - 次のステップ（実装に着手するための `/start` など）へ進むための確認を行います。
   - **重要**: `/plan` から実装フェーズへ移行する際は、直接ファイルを編集せず、必ず `/start` ワークフローを実行するか、新しい作業ブランチを作成してください。 `main` ブランチでの直接編集は厳禁です。
