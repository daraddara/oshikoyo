description: リポジトリの内容（ドキュメント、ソースコード等）や現在の状態を徹底的に調査し、ユーザーの質問に回答のみを行う（ファイル変更を行わない）
---
# 目的
本格的な開発や修正に入る「前」の段階、あるいは単なる現状確認として、リポジトリ内のコード、ドキュメント、設定、コミット履歴などを詳細に分析し、ユーザーの疑問を解消するためのワークフローです。このワークフロー内では、指示がない限りプロジェクトのファイル変更は一切行いません。

# 手順
1. **質問の意図・対象の把握**:
   - ユーザーが何を知りたいのか、どの機能やコード、ドキュメントが関係しているのかを特定します。

2. **徹底的なリポジトリ調査**:
   - `ls -R`, `view_file`, `grep_search` 等のツールを駆使し、関連するすべてのソースコード、テスト、ドキュメント、`.agents` 内のルールを読み込みます。
   - 必要に応じて、コミット履歴（`git log`）も参照し、経緯を把握します。

3. **現状の分析と整理**:
   - 収集した情報を元に、現在の実装状況、ドキュメントとの整合性、潜在的な課題などを整理します。
   - ファイルの書き換えは行わず、頭の中（あるいは一時的なメモ）で分析を完結させます。

4. **回答と提案**:
   - 調査結果を元に、ユーザーの質問に日本語で詳しく回答します。
   - **重要**: 回答の際、必要であれば「今後の修正方針案」や「`ISSUES.md` への追記案」などのテキストを提示しますが、**実際のファイルへの書き込みは行いません。**

5. **ネクストステップの確認**:
   - 回答内容に満足いただけたか確認し、もし実装が必要になった場合は `/start` や `/feature` 等の適切なワークフローへの移行を提案します。

# 禁止事項
- `write_to_file`, `replace_file_content` 等を用いた、プロジェクト本番ファイル（.js, .css, .html, .md等）の変更。
- ブランチの作成（調査のみであるため不要）。
- `npm test` による破壊的な変更の誘発。
