# PR管理ワークフロー (`/pr`)

このワークフローは、GitHub上のPRを安全かつ効率的にローカルで確認し、メインブランチへ統合するための手順を提供します。

## 1. PRの一覧と詳細確認
まずは現在のPR一覧を確認し、対象のPR番号を特定します。
```bash
gh pr list
```
特定のPRの内容を確認する場合：
```bash
gh pr view [PR番号]
```
> [!NOTE]
> 複数のPRがある場合は、この時点でどの番号を処理するか決定してください。

## 2. PRのチェックアウトと環境準備
対象のPRをローカルブランチとして取得し、最新の `main` 状態を反映させます。
```bash
gh pr checkout [PR番号]
git fetch origin main
```

## 3. コンフリクトの確認と解消
もし `main` との間に競合がある場合は、修正を行います。
// turbo
```bash
git merge origin/main
```
> [!IMPORTANT]
> コンフリクトが発生した場合は、エージェントがファイル内容を分析し、手動で修正を行ってください。修正後は `git add .` を行い、コミットを完了させてください。

## 4. 動作検証とテスト
PRの内容が正しく機能するか、既存機能に影響がないかを確認します。
```bash
npm test
npm run e2e
```
> [!TIP]
> UIの変更がある場合は、`http://localhost:8081` で起動してスクリーンショットを確認してください。

## 5. レビューと対応方針の決定
テスト結果とコード内容をレビューし、以下のいずれかの方針を決定します。

### A. コメントで指摘・提案する（原則）
問題がある場合や改善の余地がある場合は、GitHub上でコメントを残します。
```bash
gh pr comment [PR番号] --body "指摘・提案内容"
```
> [!NOTE]
> 指摘後は、作者による修正を待つか、相談を継続してください。

### B. 直接修正する（例外）
以下の場合に限り、ローカルで修正を行い commit/push します。
- ユーザーから明示的な修正指示があった場合。
- 誤字脱字の修正など、極めて軽微な内容である場合。

## 6. マージの実行
すべてが適切に解決され、マージの許可が得られたら Squash Mergeを実行します。
```bash
gh pr merge --squash --delete-branch
```

## 7. クリーンアップ
マージ完了後、最新の `main` へ戻り、ローカルを整理します。
```bash
git checkout main
git pull origin main
```
