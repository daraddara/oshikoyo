# Google Jules — 環境固有の追加指示

> このファイルは、**Google Jules** が本リポジトリで作業する際に、
> [docs/AGENTS.md](../docs/AGENTS.md) の共通ルールに**追加**して従うべき、
> Jules 固有の指示を定義します。
>
> **共通ルール（コーディング規約・テスト・Git運用等）は [docs/AGENTS.md](../docs/AGENTS.md) を参照してください。**

---

## 動作環境

Jules は **Linux（クラウド）環境**で動作する前提です。
Windows 固有の手順（PowerShell スクリプト、ブラウザプロファイル隔離等）は適用されません。

## PR 作成時のガイドライン

### UI 変更がある場合

Julesに自律的なUIテストも任せるため、UIに変更がある場合やPRを作成する前には、必ずヘッドレス環境で `npm run e2e` を実行して成功を確認してください。
また、JulesはUI変更時に必ず `tests/e2e/screenshots/` に実行結果の画像を保存し、PRの本文にその画像を貼り付けること。

以下の情報も PR の説明に含めてください：

- 変更が影響する画面・コンポーネントの名前
- 変更前後の見た目や振る舞いの差分をテキストで詳細に説明
- ダークモード・レスポンシブ表示への影響の有無
- 手動での確認手順（人間のレビュアーがどう検証すべきか）

### 外部ライブラリの提案

外部ライブラリの導入が必要な場合は、PR の説明に理由と代替案を記載してください。

## ワークスペース衛生

### ⚠️ 絶対禁止：リポジトリへのスクラッチファイルのコミット

> **過去の実例**: 夜間チェックPRにおいて、以下のファイルが繰り返しリポジトリにコミットされた。
> これはレビュアーの信頼を損なう重大なルール違反であり、PRが自動クローズされる原因となる。

以下のファイルは **`git add` しないこと・コミットに含めないこと**：

| 禁止ファイル | 理由 |
|---|---|
| `plan.md` / `*.plan.md` | タスク計画はリポジトリ管理対象外 |
| `pnpm-lock.yaml` | このプロジェクトのパッケージマネージャーは **npm**（`package-lock.json` を使用）。`pnpm` は不使用 |
| `patch_*.js` / `patch_*.ts` | パッチ適用スクリプトは一時ファイル |
| `test_*.js` / `test_xss*.js` など | `tests/` 配下の正規テスト以外のスクリプトはすべて一時ファイル |
| `.jules/` ディレクトリ | Jules 内部のメタデータ・設定フォルダ |
| `tests/e2e/*-snapshots/*.png`（新規追加分） | E2Eスナップショットを新規追加する際は事前に明示的な承認が必要 |

これらが必要な場合は `/tmp/` を使用してください。**コミット前に必ず `git status` で差分を確認し、上記ファイルが含まれていないことを検証すること。**

### 一時ファイルの扱い

作業中に生成する検証用スクリプト・ハッシュ試験ファイル・ベンチマーク・diff ファイルは **必ず `/tmp/` に作成**してください。
以下のようなファイルをリポジトリにコミットすることは**禁止**です：

- `test-*.js`（`tests/` 配下の正式テストを除く）
- `*.diff` / `patch.diff`
- `test-benchmark.js` 等のプロファイリング用スクリプト

### ベンチマークテスト

パフォーマンス計測が目的のテストは `tests/` ディレクトリに追加しないでください。
追加する場合は、`npm test` の通常スイートから除外される仕組み（別スクリプト等）を設けてください。

### 既存ファイルの削除禁止

ワークスペース衛生のために一時ファイルを削除する際、**既存の正規ファイルを巻き込まないでください**。

特に以下に注意してください：
- `tests/` ディレクトリ内のファイルは、自分が今回のタスクで新規追加したもの以外は削除しないこと
- 削除対象が「自分で追加した一時ファイル」か「既存の正規ファイル」かを `git log --follow <file>` や `git show HEAD:<file>` で確認してから削除すること
- `git status` で変更リストを確認し、想定外のファイルが含まれていないかをコミット前に必ずチェックすること

**過去の事例**: workspace hygiene ルールに従って一時ファイルを削除した際、`tests/import_benchmark.test.js`（PR#65 から存在する正規テスト）を誤って削除するコミットが発生した。

## ブランチ・PRの扱い

### ブランチの重複作成禁止

同じ修正内容の PR が既にマージ済みの場合、同一ブランチを再プッシュしたり、新規 PR を作成したりしないでください。
作業開始前に `git log origin/main` や GitHub の closed PR を確認してください。

### マージ後のブランチ削除

PR がマージされたら、対応するブランチは速やかに削除してください。
リモートに残存ブランチを放置しないでください。

## マルチエージェント協調ルール

本リポジトリでは **Claude Code（ローカル）**・**Antigravity（ローカル）**・Jules（クラウド）の3エージェントが協調して作業します。
詳細な役割分担・レビューフローは [docs/AGENTS.md](../docs/AGENTS.md) のセクション 10 を参照してください。

### 自己マージ禁止

**Jules が作成した PR を Jules 自身がマージすることは禁止です。**
PR 作成後は Claude Code または Antigravity によるレビューを待ち、マージは別エージェントまたはユーザーが行います。

PR 本文には意図・影響範囲・テスト結果を明記し、レビュアーが判断しやすい内容にしてください。
