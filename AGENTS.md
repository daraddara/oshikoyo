## 1. プロジェクト概要

| 項目 | 内容 |

## 2. 言語規約

- **すべての応答・生成ドキュメント・コミットメッセージの説明文は日本語**で記述してください。
- 英語のテンプレート見出し（"Proposed Changes" 等）は使用せず、日本語に翻訳してください。
- 専門用語（CSS プロパティ名、API名など）は英語のまま使用して構いません。

## 3. コーディング規約

### JavaScript

- モダンな ES6+ 構文（`const`/`let`, アロー関数, 分割代入, テンプレートリテラル等）を使用してください。
- DOM 操作は `getElementById` を基本とし、複雑な場合は `querySelector` を使用してください。
- 各関数には必ず **JSDoc** 形式でコメントを記述してください（説明・引数・戻り値）。
- 外部ライブラリの導入が必要な場合は、事前に提案（Antigravity: チャット / Jules: PR説明）してください。

### CSS

- 命名規則は **BEM** 形式（`block__element--modifier`）を使用してください。
- 色・サイズ等の共通値は `:root` で CSS 変数として定義し、再利用してください。
- レスポンシブ対応を前提とし、マジックナンバー（固定値）の多用を避けてください。

---

## 4. デザイン規約

### スタイル管理
- `z-index` の値は直接指定せず、CSS 変数を介して階層管理してください。
- `transition` を使用して、滑らかなインタラクションを実装してください。

### カレンダー固有の制約
- 繰り返し要素（日付セル、予定セル等）のレイアウトには **CSS Grid** を優先してください。
- 「推し」のイメージカラーを動的に変更できるクラス設計（BEM modifier）を考慮してください。
- ダークモード対応を想定し、背景色と文字色は CSS 変数をペアで定義してください。

## 5. テスト規約

### 基本原則
- `script.js` 内のロジック（祝日判定、日付計算等）を変更した場合は、必ずテストを実行してください。
- バグ修正時は、修正前にそのバグを再現するテストケースを `tests/` 内に追加してください。

### テスト実行手順

```bash
# 1. 依存パッケージの確認・インストール
npm install

# 2. テスト実行
npm test
```

- テストが失敗した場合は、原因を分析し、修正した上で再度 `npm test` を実行してください。
- **すべてのテストが 100% パスした状態**でのみコミットを行ってください。

### テストの記述スタイル
- テストファイルは `tests/` ディレクトリに `対象ファイル名.test.js` の形式で作成してください。
- `script.js` は非モジュール（ESM ではない）のため、テスト側では `fs.readFileSync` と `new Function` を使用し、特定のコメントブロック（例: `// --- Holiday Logic ---`）を基準にコードを抽出して検証してください。
- 新しい関数を追加する際は、テスト側で抽出しやすいように**明確なコメント区切り**を挿入してください。
- カレンダーロジックは**境界値**（月末、月初、閏年等）を重点的にテストしてください。
- DOM 操作を伴う関数は `jsdom` を使用してモックしてください。

### 自律実行ルール
- `script.js` または `*.test.js` を変更した後は、**必ず** `npm test` を実行して結果を確認してください。
- UIに変更を加えた場合は、`npm run e2e` を実行してスクリーンショットを含むテスト結果を報告してください。
- テストが失敗した場合は、原因を分析し、修正を行った上で再度実行してください。
- 最終的に「テストがすべてパスした状態」で作業完了としてください。

## 6. Git 運用

### ブランチ戦略
- `main` ブランチへの直接コミットは**禁止**です。
- 作業ブランチの命名規則:
  - 新機能: `feature/[機能名]`
  - バグ修正: `fix/[問題名]`
  - リファクタリング: `refactor/[対象]`

### コミット規約
- コミットメッセージは [Conventional Commits](https://www.conventionalcommits.org/) 形式に従ってください。
- コミット前に必ず `npm test` を実行し、全テストがパスしていることを確認してください。

### マージの条件 (Definition of Done)
以下をすべて満たさない限り、マージを提案しないでください:

1. すべてのロジックテスト（Vitest）が 100% パスしていること
2. 修正したコードに JSDoc 等のドキュメントが付与されていること
3. `ISSUES.md` に関連する修正の場合、該当項目のステータスが更新されていること

### マージ戦略
- **Squash Merge** を使用してください。
- マージコミットメッセージは変更の要約としてください（例: `feat: 祝日ロジックの保守性を向上 (ISSUES A-01)`）。
- `main` ブランチの履歴は常にリニア（一本道）を維持してください。

## 7. ドキュメント管理

### Single Source of Truth
- コーディング規約・Git 運用・テスト手順などの「技術的ルール」は、このファイルと各エージェント固有のファイルにのみ記述してください。
- ルートの Markdown ファイル（README, TESTING, DEVELOPMENT 等）には、上記ルールの詳細を重複して記載しないでください。

### 各ファイルの保守指針

| ファイル | 記述範囲 |
|---|---|
| `README.md` | プロジェクトの目的、主要機能、技術スタック概要 |
| `DEVELOPMENT.md` | 環境構築の初歩的手順のみ |
| `TESTING.md` | テストの理念と対象（何をテストするか） |
| `ISSUES.md` | 仕様の曖昧点、未着手の機能案（常に最新に保つ） |

### コード変更時の自動更新ルール

**コード変更の完了前に、以下のチェックリストを必ず確認してください：**

- [ ] 機能を新規追加した場合 → `README.md` の「特徴」セクションを更新したか
- [ ] 仕様の曖昧さが解消された場合 → `ISSUES.md` の該当項目を「完了」に移動したか
- [ ] テスト方針に変更がある場合 → `TESTING.md` を更新したか

## 8. 機能追加ワークフロー

新しい機能やコンポーネントを追加する際は、以下の手順に従ってください：

1. `index.html` の適切な位置に HTML 構造を追加する
2. `style.css` に必要なスタイルを追記する（BEM 規則に従うこと）
3. `script.js` にロジックを実装する（JSDoc を付与すること）
4. `npm test` を実行し、すべてのテストがパスすることを確認する
5. UI に変更がある場合は、変更内容を記録する（環境に応じた方法で）
6. 関連する `README.md` 等のドキュメントを更新する
